# Research: Automatic Request Tracing via LiteLLM → Langfuse Integration

**Feature**: [spec.md](spec.md) | **Plan**: [plan.md](plan.md)
**Date**: 2026-04-18

## 1. LiteLLM Callback Architecture

### Decision
Use LiteLLM's **post-request callback mechanism** to capture trace data after model completion but before response is returned to client.

### Rationale
LiteLLM supports Python callables registered in config that receive:
- Model name, prompt, response, tokens, latency
- Custom metadata passed through model parameters
- Errors/exceptions for failure tracing

Post-request callback allows:
- Complete request/response pair capture
- Token counts from completed inference
- Non-blocking trace submission (async capable)
- Metadata extraction from request context

### Alternatives Considered
- **Pre-request callback**: Only captures request intent, not response data → rejected (incomplete traces)
- **Middleware wrapper**: Adds extra network layer → rejected (unnecessary complexity)
- **LLM log parsing**: Fragile, parsing-dependent → rejected (not recommended)

### Implementation Approach
```python
# config/litellm_langfuse_callback.py
async def langfuse_callback(kwargs, response, start_time, end_time):
    # Extract prompt, response, model, tokens, latency
    # Extract metadata from kwargs.get('metadata')
    # Submit trace to Langfuse asynchronously
    # Return without blocking request path
```

## 2. Trace Schema Definition

### Decision
Langfuse trace schema includes the following mandatory fields:

```json
{
  "trace_id": "uuid",                        // Generated by LiteLLM or Langfuse
  "correlation_id": "uuid|request_id",      // From client request header or generated
  "timestamp": "ISO8601",                    // Request start time
  "prompt": "string",                        // Full prompt text
  "response": "string",                      // Model response
  "model_name": "string",                    // e.g., "mistral-7b-instruct"
  "tokens": {
    "input": integer,
    "output": integer,
    "total": integer
  },
  "latency_ms": {
    "queue": integer,          // Time in gateway queue
    "inference": integer,      // vLLM inference time
    "total": integer           // Total request latency
  },
  "metadata": {
    "team_id": "string",
    "user_id": "string",
    "api_key": "string (hashed)",
    "use_case": "string|null",
    "request_id": "string"
  },
  "status": "success|error",
  "error": "string|null"
}
```

### Rationale
This schema captures:
- Cost attribution (tokens, model)
- Audit trail (team_id, user_id, api_key)
- Performance debugging (latency breakdown)
- Request correlation (trace_id, correlation_id)
- Observability (status, error handling)

### Alternatives Considered
- **Minimal schema** (only prompt/response/tokens): Too limited for cost tracking and debugging → rejected
- **Flat structure** (no nested objects): JSON parsing harder in dashboards → rejected

## 3. Metadata Propagation Pattern

### Decision
Metadata flows via HTTP headers from client → LiteLLM → Langfuse trace.

**Client Headers (required)**:
```
X-Team-Id: {team_id}
X-User-Id: {user_id}
X-Api-Key: {api_key_value}
X-Use-Case: {use_case}  # optional
X-Correlation-Id: {correlation_id}  # optional, generated if missing
```

**LiteLLM Processing**:
1. Extract headers from request
2. Validate team_id, user_id, api_key (authentication)
3. Pass metadata to model via `metadata` parameter in request kwargs
4. Langfuse callback reads metadata from kwargs and trace context

**Langfuse Trace**:
- All metadata fields copied into trace JSON
- API key stored as salted hash (not plain text)

### Rationale
Header-based propagation:
- Follows HTTP best practices (metadata in headers, not body)
- Widely supported across proxies and load balancers
- Cleanly separates metadata from prompt content
- Enables easy filtering and correlation in dashboards

### Alternatives Considered
- **Request body metadata**: Conflicts with prompt structure → rejected
- **Bearer token extraction**: Only supports API key, not team/user context → rejected

## 4. Correlation ID Strategy

### Decision
Use UUID v4 for correlation IDs, generated by client or LiteLLM if missing.

**Flow**:
1. Client generates UUID or uses transaction ID
2. Send as `X-Correlation-Id` header
3. LiteLLM validates format, generates if missing
4. Passed to vLLM in request headers
5. Included in Langfuse trace

### Rationale
UUID v4:
- Guaranteed uniqueness across distributed systems
- No coordination required
- Standard practice in observability (OpenTelemetry, Jaeger)
- Easy to search in dashboards

### Alternatives Considered
- **Request hash**: Non-deterministic for same request → rejected
- **Timestamp-based**: Risk of collisions under high load → rejected

## 5. Retry & Backoff Policy

### Decision
Use exponential backoff with jitter for Langfuse trace submission failures.

**Strategy**:
```
max_retries = 3
base_delay = 100ms
max_delay = 5s

delay = min(base_delay * (2 ^ attempt) + random(0, 100ms), max_delay)
```

**Behavior**:
- Attempt 1: ~100ms delay (instant on first retry)
- Attempt 2: ~200-300ms delay
- Attempt 3: ~400-600ms delay
- After 3 failures: buffer to disk/queue for async resubmission

### Rationale
- Avoids thundering herd on Langfuse restart
- Jitter prevents synchronized retries from multiple gateways
- 3 retries + backoff covers transient failures (network blips, brief outages)
- Disk buffer ensures traces aren't lost during extended Langfuse downtime

### Alternatives Considered
- **No retry**: Loses traces on transient errors → rejected
- **Fixed delay**: Can amplify thundering herd → rejected
- **Immediate retry**: Wastes resources on persistent failures → rejected

## 6. Buffering & Resilience

### Decision
Implement **file-based trace buffer** on LiteLLM for traces that fail after retries.

**Architecture**:
```
LiteLLM callback
  ↓
Attempt submit to Langfuse (with retries)
  ↓
Success? → Done
  ↓
Failure after retries? → Append to disk buffer
  ↓
Async buffer flusher (periodic, e.g., every 60s):
  - Read buffered traces from disk
  - Retry submission to Langfuse
  - On success, delete from buffer
  - On failure, keep in buffer (exponential backoff)
```

**Buffer management**:
- Location: `litellm/data/trace_buffer.jsonl` (newline-delimited JSON)
- Max size: 100MB (configurable)
- Retention: until successfully submitted
- Startup: Attempt to flush buffer on LiteLLM startup

### Rationale
- Traces are not lost during Langfuse maintenance windows
- No external dependencies (Redis not required initially)
- Scalable for small deployments (RTX 3060 node)
- Easy to debug (human-readable JSONL format)

### Alternatives Considered
- **Redis queue**: Adds operational complexity, dependency → rejected for v1
- **In-memory buffer**: Traces lost on LiteLLM restart → rejected
- **Drop failed traces**: Violates observability principle → rejected

## 7. Token Extraction from vLLM

### Decision
Extract token counts from vLLM response metadata in LiteLLM callback.

**Assumption**: vLLM returns response in format:
```json
{
  "id": "request-id",
  "object": "text_completion",
  "created": 1234567890,
  "model": "mistral-7b-instruct",
  "choices": [{"text": "response", "finish_reason": "stop"}],
  "usage": {
    "prompt_tokens": 42,
    "completion_tokens": 18,
    "total_tokens": 60
  }
}
```

**Extraction logic** (in callback):
```python
usage = response.get("usage", {})
input_tokens = usage.get("prompt_tokens", 0)
output_tokens = usage.get("completion_tokens", 0)
total_tokens = usage.get("total_tokens", 0)
```

### Rationale
- vLLM OpenAI-compatible API includes usage in response
- No additional vLLM queries needed
- Reliable and standard format

### Requirements
- Requires vLLM with OpenAI-compatible API (v0.3+)
- If using older vLLM versions, adjust token_extractor.py logic per troubleshooting in quickstart.md

### Note
If vLLM response format differs, update extraction logic in research v2.

## 8. Industry Best Practices Reference

### Sources
- **Kong API Gateway**: Uses plugin system for observability, supports callbacks and async event submission
- **API7 Gateway**: OpenTelemetry integration at gateway layer, traces without SDK in application code
- **Tyk API Gateway**: Middleware-based observability, supports custom drivers for trace backends
- **OpenTelemetry**: Standardized instrumentation at proxy/gateway layer to avoid application-code bloat

### Takeaway for This Design
Gateway-level tracing (LiteLLM callbacks) is the industry standard for observability at scale, avoiding SDK scatter across application code and ensuring consistent, complete trace coverage.

## Design Decisions Summary

| Area | Decision | Rationale |
|------|----------|-----------|
| Callback Timing | Post-request (async) | Complete data + non-blocking |
| Trace Schema | Comprehensive (18+ fields) | Cost attribution + audit + debugging |
| Metadata Flow | HTTP headers → LiteLLM kwargs → Langfuse trace | Standard HTTP practices |
| Correlation ID | UUID v4 generated on client/LiteLLM | Uniqueness + distributed systems ready |
| Retry Strategy | Exponential backoff with jitter, 3 attempts | Handles transients + prevents thundering herd |
| Buffering | File-based (JSONL) on LiteLLM | Resilient + simple + debuggable |
| Token Extraction | From vLLM response usage field | Reliable + no extra queries |
